# SmartVoyage 项目问题总结

这份文档只保留当前主线真正有代表性的工程问题，方便后续复盘和面试表达。

## 1. 过度服务拆分，导致复杂度高于业务本身

### 问题现象

项目早期按小功能拆了过多服务，天气、票务、订单、旧 `travel_plan` 等都各自独立，导致：

- 调试链路长
- 服务数量多
- 错误定位慢
- 很多问题不是业务问题，而是编排和协议问题

### 根因

- 设计时把“多智能体展示效果”放在了“可维护性”前面
- 一个小能力一个服务，粒度过细
- 读能力和副作用能力没有按职责收束

### 解决方案

- 重构为 `2` 个 A2A + `2` 个 MCP
- 统一只读链路到 `TravelDecisionAgent`
- 统一订单生命周期到 `TransportOrderAgent`
- 删除酒店、景点、旧 `travel_plan` 主线

### 结果

- 主线更容易理解
- 日常联调成本显著下降
- 复杂度开始和真实业务规模匹配

## 2. A2A 承载层不统一，接口和日志风格分裂

### 问题现象

早期 A2A 与 MCP 承载方式不一致，导致：

- 协议风格不统一
- 健康检查、元信息、调用方式不一致
- 日志和错误处理各自为政

### 根因

- 服务承载层缺少统一抽象
- 过早引入多种承载方式，增加了排查成本

### 解决方案

- A2A 主线迁移到 `FastAPI / ASGI`
- MCP 主线也统一走 `FastAPI / ASGI`
- A2A 统一暴露：
  - `GET /health`
  - `GET /metadata`
  - `POST /invoke`
- 引入统一协议对象和中间件：
  - `utils/service_protocol.py`
  - `utils/fastapi_middleware.py`

### 结果

- 联调方式统一
- 健康检查和元信息更清晰
- 后续扩展和测试更容易标准化

## 3. 同步工作流跑在异步服务里，导致真实请求 500

### 问题现象

`TransportOrderAgent` 迁到 FastAPI 后，手工联调时出现真实 `500`：

- 查单失败
- 自动订票失败
- 日志出现 `asyncio.run() cannot be called from a running event loop`

### 根因

- 外层服务已经是异步接口
- 但订单域工作流仍按同步方式调用
- 节点内部又嵌套了 `asyncio.run(...)`

### 解决方案

- `TransportOrderAgent` 改成真正的异步 LangGraph 工作流
- 外层改用 `await workflow.ainvoke(...)`
- 节点统一改成 `async def`
- 节点内部全部改成直接 `await ...`

### 结果

- `查询我的订单` 恢复正常
- `transport_decision -> 自动下单` 全链路恢复
- 退票和改签链路也能真实联调通过

## 4. 没有统一 request_id，跨服务日志难串起来

### 问题现象

编排层、A2A、MCP 分别打日志时，很难定位同一轮请求到底经过了哪些服务、在哪一步失败。

### 根因

- 缺少统一请求上下文
- 日志内容没有固定字段

### 解决方案

- 引入 `request_id`
- 在 orchestrator、A2A、MCP 之间透传
- 在日志中过滤注入：
  - `request_id`
  - `domain`
  - `action`

### 结果

- 现在能从一条请求串到完整调用链
- 排查真实联调问题时效率明显提高

## 5. Prompt 容易越权，模型会补出后端并未确认的业务细节

### 问题现象

复杂任务里，模型容易直接给出过于具体的订票细节，甚至在后端尚未确认库存前就表现得像“已经可以下单”。

### 根因

- Prompt 负责的事情过多
- 缺少后端强校验边界
- 规划和执行之间的责任划分不够清晰

### 解决方案

- 统一采用“结构化抽取 + 后端校验 + 缺字段追问”的模式
- Prompt 只负责：
  - 意图识别
  - 结构化抽取
  - 建议生成
- 后端负责：
  - 最小进入条件
  - `pending_context`
  - 唯一命中
  - 库存一致性
  - 状态流转

### 结果

- 复杂任务结果更可预测
- 大模型不再承担业务真相来源

## 6. 只靠手工对话回归，容易反复修 bug

### 问题现象

如果只靠 CLI 手测，很多修复在下一轮重构里又会被破坏，而且很难快速知道是意图、路由还是文案回退了。

### 根因

- 缺少自动回归
- 没有固定基线样例

### 解决方案

- 当前先补最小 `LangSmith` 基线集
- 使用单一基础数据集，同时覆盖：
  - 时间查询
  - 天气查询
  - 火车票查询
  - 机票查询
  - `transport_decision`
  - 直接下单
  - 退票
  - 改签
- evaluator 固定检查：
  - intent
  - route
  - response keywords
  - pending context
  - DB state

### 结果

- 后续改 orchestrator、prompt、A2A 时，可以更快发现回归
- 项目开始从“手工调 Demo”转向“有最小自动回归的工程项目”

## 7. 跨服务只传自然语言文本，复杂任务容易变脆

### 问题现象

`transport_decision` 和订单创建链路里，前一层服务经常把结果先写成自然语言，再由后一层服务继续理解。这样虽然容易演示，但一旦文案措辞变化，后续行为就可能漂移。

### 根因

- 服务协议只有 `text`
- 编排层和订单域都需要从上游文案里再次恢复业务事实
- 这会把“模型表述变化”错误放大成“执行链路不稳定”

### 解决方案

- 把 A2A 响应协议升级成双通道：
  - `text`
  - `data`
  - `meta`
- `TravelDecisionAgent` 现在除了返回自然语言结果，还会返回结构化天气 / 时间 / 票务 payload
- orchestrator 和订单域优先消费结构化 payload，再把自然语言结果留给用户展示

### 结果

- 复杂任务不再主要依赖文案桥接
- 下单路径更稳定
- 调试时可以直接看结构化 payload，而不是从自然语言里猜上游到底返回了什么

## 8. 高风险副作用缺少真正暂停点，CLI 下很难做人审

### 问题现象

在接入自动下单、退票、改签后，系统虽然能执行，但高风险动作没有真正的暂停点。即使想做人审，也只能在业务层手写一轮“确认吗”，并不能真正暂停和恢复工作流。

### 根因

- 之前只有 `pending_context`
- 没有使用 LangGraph 的 checkpoint / interrupt / resume
- CLI 缺少审批线程概念

### 解决方案

- `TransportOrderAgent` 接入 `InMemorySaver`
- 在 `create_order / cancel_order / change_order` 之前统一进入 `review` 节点
- `review` 节点使用 `interrupt(...)` 返回审批 payload
- CLI 通过 `yes / no` 恢复或取消执行

### 结果

- 命令行下也能做真正的 HITL
- 工作流不再是假暂停，而是 LangGraph 原生暂停后恢复
- 当前限制是 checkpoint 仍为进程内存级，服务重启后审批线程会丢失

## 面试时建议优先讲的 4 个问题

1. 我把过度拆分的多服务结构收敛成 `2` 个 A2A + `2` 个 MCP，降低了系统复杂度。
2. 我把 A2A 主线迁到 `FastAPI / ASGI`，统一了协议、日志和联调方式。
3. 我修过一个典型的异步工程问题：同步工作流跑在异步服务里，导致 `asyncio.run()` 冲突和真实 `500`。
4. 我把大模型的职责限制在结构化抽取和建议生成，库存一致性和状态流转全部收回后端做确定性控制。
