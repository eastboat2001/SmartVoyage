# SmartVoyage 项目问题总结

这份文档用于总结 SmartVoyage 项目开发和改造过程中遇到的典型问题、根因分析、解决方案，以及最终效果。  
面试时可以直接从这里挑 2 到 4 个问题展开讲，不需要全部背诵。

## 1. 多智能体看起来像功能拆分，不像协作系统

### 问题现象

项目最初虽然有天气查询、票务查询、票务预定三个 Agent，但它们更像被前端路由到不同功能模块，没有形成真正的协作链路。

### 根因

- 入口层只是做意图识别后分发，没有统一编排。
- Agent 之间缺少明确的依赖关系。
- 没有一个场景能体现“一个 Agent 的结果影响下一个 Agent 的决策”。

### 解决方案

- 新增统一编排层，放在 `utils/orchestrator.py`。
- 新增 `travel_plan` 意图，支持“先查天气，再判断出行方式，再查票，必要时再订票”的协作链路。
- 使用结构化输出 `TravelPlanResult` 约束规划阶段输出，避免跨 Agent 协作过程失控。

### 结果

- 系统不再只是把请求路由给不同服务，而是具备了跨 Agent 决策和串联能力。
- 面试时可以明确说明：这个项目不只是多服务，而是一个带编排逻辑的多智能体协作系统。


## 2. 结构化输出偶尔失败，导致链路中断

### 问题现象

意图识别、天气 SQL 生成、票务 SQL 生成都依赖结构化输出。如果模型输出漂移，整个流程会直接失败。

### 根因

- 大模型输出受提示词、上下文、模型能力影响，不能假设每次都稳定符合 schema。
- 原始实现对结构化输出失败缺少重试和兜底。

### 解决方案

- 把结构化输出统一封装到 `utils/resilient_llm.py`。
- 增加结构化输出重试次数配置：`SMARTVOYAGE_STRUCTURED_RETRY_COUNT`。
- 在天气 Agent、票务 Agent、入口编排层统一接入容错调用。
- 失败时不直接崩溃，而是回退到用户可理解的追问或错误提示。

### 结果

- 系统对模型输出波动更稳健。
- 结构化输出从“单点脆弱依赖”变成“带重试和兜底的基础能力”。


## 3. 主模型不可用时，整个系统没有降级能力

### 问题现象

如果主模型 API key 无效、接口异常，或者当前 provider 不可用，原系统会整体失效。

### 根因

- 模型接入只有单 provider。
- 没有主模型失败后的回退方案。

### 解决方案

- 在 `config.py` 中增加 fallback 相关配置：
  - `SMARTVOYAGE_FALLBACK_PROVIDER`
  - `SMARTVOYAGE_FALLBACK_MODEL_NAME`
  - `SMARTVOYAGE_FALLBACK_BASE_URL`
  - `SMARTVOYAGE_FALLBACK_API_KEY`
- 在 `utils/resilient_llm.py` 中实现 primary -> fallback 的模型调用策略。
- 在普通文本生成、结构化输出、订票 Agent 构造时统一使用这套回退逻辑。

### 结果

- 主模型失败时，系统仍有机会继续完成请求。
- 这个改造更贴近真实线上系统，而不是单模型 Demo。


## 4. Agent 服务在线，但业务上依然可能失败

### 问题现象

最初测试时，A2A agent card 可以拿到，说明服务在线，但用户请求仍然返回失败或“天气不可用”。

### 根因

- “服务在线”只说明端口和 agent card 正常，不代表业务数据一定存在。
- 原编排层把“无数据”和“服务不可用”混在一起处理，导致错误文案误导用户。

### 解决方案

- 在 `utils/orchestrator.py` 中区分三种状态：
  - `completed`
  - `failed`
  - `no_data`
- 只有真正失败时才提示“服务不可用”或“降级”。
- 如果只是数据库没有命中结果，则提示“未找到天气数据”或“未找到票务数据”。

### 结果

- 系统对用户的反馈更准确。
- 面试时可以说明：我不仅做了异常处理，还区分了“系统故障”和“业务无数据”这两类完全不同的问题。


## 5. 演示数据过期，导致“明天/后天”这类问题经常查不到结果

### 问题现象

数据库中的天气和票务数据日期比较旧，测试“明天”“后天”时经常返回空结果，影响项目演示效果。

### 根因

- 初始 SQL 数据是静态的。
- 演示问题使用相对日期，但数据库并不保证一直覆盖这些日期。

### 解决方案

- 直接重写 `sql/insert_data.sql`，替换成一批新的可演示数据。
- 当前数据重点覆盖：
  - 天气/票务：`2026-03-21` 到 `2026-03-25`
  - 演唱会：`2026-04`
- README 中改为推荐使用绝对日期进行演示，避免相对日期和数据库错位。

### 结果

- 演示稳定性明显提升。
- 可以提前准备“100% 命中”的测试问题，减少现场翻车概率。


## 6. 跨 Agent 订票链路会臆造细节，导致查票和订票失败

### 问题现象

在 `travel_plan` 场景下，模型有时会在没有看到真实票务结果前，提前生成过于具体的订票要求，比如：

- 指定不存在的车次
- 指定数据库里没有的站点信息
- 指定过细的时间窗口

最终导致票务查询和订票都失败。

### 根因

- 规划阶段承担了过多职责。
- 订票 query 在没有真实票务结果的情况下就被模型提前生成。

### 解决方案

- 精简 `TravelPlanResult`，去掉提前生成的 `order_query`。
- 修改提示词，明确禁止模型虚构车次、航班号、站点等未确认信息。
- 改成两阶段流程：
  1. 先做天气判断和票务查询
  2. 只有查到真实票务结果后，才基于真实结果构造订票请求
- 如果前一步没查到票，则不继续提交订票请求。

### 结果

- 协作链路更稳，逻辑更符合真实业务。
- 可以向面试官说明：我发现了模型在多阶段任务中的“幻觉放大”问题，并通过流程拆分解决了它。


## 7. 超时和下游故障会把用户会话直接打断

### 问题现象

某个 Agent 或 MCP 服务超时、异常时，原来很容易直接把错误抛到用户侧。

### 根因

- 原流程缺少统一的超时控制。
- 没有针对不同下游服务的清晰兜底文案。

### 解决方案

- 在编排层中增加 `asyncio.wait_for` 做 Agent 超时控制。
- 在 `config.py` 中增加 `SMARTVOYAGE_AGENT_TIMEOUT_SECONDS` 配置。
- 针对天气、票务、订票分别设计不同的兜底文案。
- 如果订票服务超时，明确说明“当前没有完成实际下单”，避免误导用户。

### 结果

- 用户体验更稳定。
- 面试时可以强调：异常路径处理是项目重点之一，而不是只做 happy path。


## 8. README 和代码状态容易不一致

### 问题现象

项目在持续修改后，README 中的示例、测试问题、待改进项容易与当前代码实现脱节。

### 根因

- 文档没有随着代码一起维护。
- 演示项目对“示例是否能跑通”非常敏感。

### 解决方案

- 同步更新 `README.md` 中的数据库日期范围、推荐测试问题、当前能力说明。
- 删除已经完成的“待改进项”，保留真正还没做的内容。
- 把可演示的问题固定成数据库里一定命中的绝对日期。

### 结果

- 项目描述、代码实现、演示数据三者保持一致。
- 面试时更容易形成完整闭环。


## 面试时建议重点讲的 4 个问题

如果面试官只问“项目里遇到过什么问题，怎么解决”，建议优先讲下面 4 个：

1. 多智能体最开始只是功能拆分，不像协作系统，所以我加了统一编排层和 `travel_plan` 协作链路。
2. 结构化输出不稳定，我做了统一重试和兜底，并支持 fallback provider。
3. 业务无数据和系统故障最开始混在一起，我把状态拆开处理，避免错误反馈误导用户。
4. 跨 Agent 订票链路里模型会提前臆造车次和站点，我改成先查真实票，再决定是否继续订票。


## 面试回答模板

可以按下面这个结构回答：

> 这个项目里我遇到的核心问题不是单点 bug，而是 AI 系统常见的工程稳定性问题。  
> 比如一开始多智能体只是功能拆分，不像真正协作，我后来加了统一编排层，把天气、票务和订票串成了一条完整链路。  
> 另外结构化输出和模型调用也不是每次都稳定，所以我做了重试、超时控制和 fallback provider。  
> 还有一个比较典型的问题是，大模型在订票链路里会臆造具体车次，导致查不到票。我最后把流程拆成“先查真实票，再继续订票”，把这个问题解决了。  
> 这些改造让我这个项目从一个可运行 Demo，更接近一个可解释、可演示、可容错的 AI 应用。
