# SmartVoyage 面试讲稿

## 1. 30 秒版本

我做的是一个面向交通票务场景的多代理 Agent 系统，叫 SmartVoyage。它支持天气查询、票务查询、下单、查单、退票和改签。我没有把它做成单纯的聊天机器人，而是重点把它设计成一个可控的事务型 Agent 系统，里面包含 Supervisor 编排、MCP 工具边界、LangGraph 状态机、HITL 审批恢复、分模型路由，以及基于 LangSmith 的性能和回归评测。

## 2. 1 分钟版本

SmartVoyage 的核心目标是解决“查询 + 决策 + 执行”混合任务。比如用户问“明天从北京到上海坐高铁还是飞机更合适，如果合适就帮我下单”，系统先做意图识别，再查询天气和票务，再进行交通决策，最后在满足条件时进入订单链路。

架构上我用了一个 Supervisor 和两个本地子代理：

- TravelReadSubagent 负责时间、天气、票务查询
- OrderSubagent 负责下单、查单、退票、改签

工具层通过 MCP 暴露，订单链路用 LangGraph 处理状态流转和 HITL。后面我又重点做了性能优化，包括减少冗余 LLM 调用、把 summary 改成 deterministic formatting、加 Redis 缓存、增加 phase metrics、做轻重模型分流和 fallback。最后用 LangSmith 跑基线回归，验证时延和 Token 确实下降，同时保证功能不回归。

## 3. 3 分钟版本

这个项目我主要做了四件事。

第一，做架构分层。我把系统拆成 Supervisor、子代理、MCP、Skill Runtime、LangGraph 和模型调用层。Supervisor 只负责路由和编排，不直接处理所有细节。这样业务逻辑不会堆成一个大 Prompt。

第二，处理事务型任务。很多 Agent 项目只做检索或问答，但这个项目里有下单、退票、改签，这些操作天然有副作用，所以我专门做了 OrderSubagent，并用 LangGraph + checkpoint + HITL 兜住高风险链路。

第三，做性能优化。我发现瓶颈不在数据库，而在 LLM 往返次数，所以我把 `transport_decision` 里的重复判定合并了，去掉了一些独立的总结型调用，给只读链路加了 Redis 缓存，还补了阶段级 metrics。后面又做了按 phase 的轻重模型分流，让低风险结构化任务走轻模型，失败时自动回退主模型。

第四，做评测闭环。我不是只看“能不能跑通”，而是用 LangSmith 追踪回归，关注 P50、P95、P99、Token 消耗、semantic match、route match。这样优化是可验证的，而不是主观感觉变快了。

## 4. 为什么只拆成两个子代理

### 标准回答

我当时没有按天气、票务、订单查询、票务操作继续拆很多 agent，而是先收敛成两个子代理：一个只读，一个事务。原因是当前项目的复杂度核心不在业务名词数量，而在执行语义。

只读链路的特点是：

- 可缓存
- 无副作用
- 适合低时延优化

事务链路的特点是：

- 有状态
- 有副作用
- 需要审批和恢复

所以按“读 / 写”来拆，边界更清晰，也更容易做性能优化和可靠性控制。

### 面试官继续追问时可以补充

如果业务继续扩展到酒店、景点、攻略、路线规划等多个新域，我会考虑再细拆 agent。当前两子代理不是能力不够，而是针对当前业务范围做的复杂度控制。

## 5. 为什么不用单 Agent

如果做成单 Agent，前期实现会更快，但后面会出现三个问题：

- 查询和事务逻辑混在一起，Prompt 很快膨胀
- 有副作用操作缺少清晰边界，风险控制困难
- 性能优化和问题定位会越来越难

所以我选择用 Supervisor + 两个子代理，把系统做成可维护、可观测、可演进的结构。

## 6. 为什么要做分模型路由

因为不同任务对模型能力的要求并不一样。

例如：

- `intent_recognition`
- `weather_plan`
- `ticket_plan`
- 日期归一化

这些任务输入短、输出结构固定、可做 schema 校验，更适合用轻模型。而像 `decision_plan`、订单复杂抽取、最终执行判断，这类任务失败成本更高，更适合保留强模型。

所以我没有盲目“全量换小模型”，而是做按阶段白名单灰度，并保留 fallback。这样既能降时延和 Token，又能控制功能风险。

## 7. 你做的最有技术含量的点是什么

推荐回答：

我觉得最有技术含量的不是单个功能，而是把这个项目做成了一个有工程闭环的 Agent 系统。也就是我不仅做了 Supervisor、多代理、事务链路和 HITL，还把性能优化、模型路由、缓存和 LangSmith 回归评测串成了一套可以持续迭代的体系。

## 8. 如果面试官说“这不就是个聊天机器人吗”

推荐回答：

我不把它定义成聊天机器人，而是一个窄领域的事务型 Agent 系统。聊天机器人主要是生成回复，而这个项目里真正复杂的是如何安全地执行查询、决策和有副作用的订单操作，并且还能做性能优化和回归验证。所以它更接近一个有工作流、有状态管理、有工具边界的 Agent 应用，而不是单纯对话系统。

## 9. 如果面试官问“项目是不是太浅”

推荐回答：

这个项目的业务广度确实是刻意收敛的，我没有一开始做成酒店、景点、攻略一体化平台。但技术深度并不浅，因为我把重点放在了多代理编排、事务状态管理、HITL、分模型路由、性能优化和评测闭环上。也就是说，它是一个窄领域深打的项目，而不是宽功能堆叠的 Demo。

## 10. 如果面试官问“为什么不用更多 agent”

推荐回答：

业务少的时候，拆太多 agent 会制造复杂度；业务多的时候，拆分是在管理复杂度。当前项目主要就两类能力：查和办，所以两子代理是比较合适的平衡点。如果后续加入酒店、景点、攻略、路线规划等异质性更强的域，我会再演进成更细粒度的多 agent 架构。

## 11. 技术亮点可直接背诵

- 用 Supervisor + 两个本地子代理，把只读查询和事务执行分开
- 用 MCP 统一工具边界，让 Agent 不直接耦合底层数据访问
- 用 LangGraph + HITL + checkpoint 承接下单、退票、改签等高风险流程
- 用 Skill Runtime 做本地 Prompt 管理，避免业务代码里到处散落模板
- 用 Redis + deterministic formatting + phase metrics 做性能优化
- 用轻重模型分流和 fallback 做成本与可靠性的平衡
- 用 LangSmith 回归体系验证时延、Token 和功能稳定性

## 12. 一句话收尾

这个项目最值得讲的不是“我做了多少功能”，而是“我把一个窄领域 Agent 系统做成了可分层、可优化、可评测、可解释的工程化系统”。
