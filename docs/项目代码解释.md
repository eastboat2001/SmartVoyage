补充：
`_call_agent()` 本身其实不直接“调用大模型”，它做的是一层项目内的统一代理分发。

你看到的 [_call_agent()]( /e:/Workstudy/projectstudy/04_Project/02_AIProject/Agent/SmartVoyage/04_Code/sh01_agent/SmartVoyage/agents/supervisor.py#L726 ) 流程是：

1. 先用 [_build_agent_chat_history()]( /e:/Workstudy/projectstudy/04_Project/02_AIProject/Agent/SmartVoyage/04_Code/sh01_agent/SmartVoyage/agents/supervisor.py#L925 ) 组装传给子代理的上下文
2. 根据 `agent_name` 分发
   - `TravelReadSubagent` -> `self.travel_read_agent.invoke(...)`
   - `OrderSubagent` -> `self.order_agent.invoke(...)`
3. 把子代理返回的 `LocalAgentResponse` 包装成 `AgentExecutionResult`

所以它不是 LangChain 那种“Supervisor 直接拿一个 agent 对象然后 `agent.invoke()`”，而是“Supervisor 调项目自定义的子代理接口”。

你之所以没看到标准 `langchain/langgraph agent.invoke()`，原因有两个。

第一，这个项目自己封装了一层“本地代理协议”。
在 [_call_agent()]( /e:/Workstudy/projectstudy/04_Project/02_AIProject/Agent/SmartVoyage/04_Code/sh01_agent/SmartVoyage/agents/supervisor.py#L726 ) 里，Supervisor 调的不是 LangChain 原生 agent，而是两个业务子代理的 `invoke()`：
- [TravelReadSubagent.invoke()]( /e:/Workstudy/projectstudy/04_Project/02_AIProject/Agent/SmartVoyage/04_Code/sh01_agent/SmartVoyage/agents/travel_read.py#L457 )
- [OrderSubagent.invoke()]( /e:/Workstudy/projectstudy/04_Project/02_AIProject/Agent/SmartVoyage/04_Code/sh01_agent/SmartVoyage/agents/order.py#L862 )

这两个 `invoke()` 是项目自己定义的同步入口，入参是 `LocalAgentRequest`，出参是 `LocalAgentResponse`。也就是说，这里做了“框架接口 -> 项目统一接口”的适配。

第二，不同子代理内部根本不是同一种实现方式。

`TravelReadSubagent` 不是标准 Agent Executor 风格。它内部在 [TravelReadSubagent.ainvoke()]( /e:/Workstudy/projectstudy/04_Project/02_AIProject/Agent/SmartVoyage/04_Code/sh01_agent/SmartVoyage/agents/travel_read.py#L407 ) 里是：
1. `infer_kind()` 判断是时间/天气/票务
2. `generate_weather_plan()` 或 `generate_ticket_plan()` 用结构化 LLM 做参数抽取
3. 再调用 MCP 工具查数据

- [ResilientModelInvoker.invoke_structured()]( /e:/Workstudy/projectstudy/04_Project/02_AIProject/Agent/SmartVoyage/04_Code/sh01_agent/SmartVoyage/llm/resilient_llm.py#L71 )
