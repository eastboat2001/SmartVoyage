# SmartVoyage 测试用例清单

这份文档用于后续做集中回归测试，重点覆盖当前项目的核心能力：

- 天气查询
- 交通票务查询
- 交通订单生命周期
- 酒店查询与酒店订单生命周期
- `travel_plan` 联动规划
- 多轮补参与 `pending_context`

默认前置条件：

- 已正确导入 `sql/create_table.sql` 与 `sql/insert_data.sql`
- 当前默认用户为 `demo_user`
- 后端服务已启动
- 若使用开发模式，推荐：

```powershell
.\.venv\Scripts\python.exe run_all.py --dev-reload
```

- CLI 单独启动：

```powershell
.\.venv\Scripts\python.exe main.py
```


## 1. 基础查询

### TC-001 天气查询

- 输入：
  - `查询2026-03-21上海的天气`
- 预期路由：
  - `WeatherQueryAssistant`
- 预期结果：
  - 返回上海 `2026-03-21` 的天气信息
  - 不应触发订单或酒店域

### TC-002 高铁查询

- 输入：
  - `查询2026-03-21北京到上海的高铁票`
- 预期路由：
  - `TicketQueryAssistant`
- 预期结果：
  - 返回北京到上海的高铁票结果
  - 不应误路由到 `order`

### TC-003 机票查询

- 输入：
  - `查询2026-03-21北京到上海的机票`
- 预期路由：
  - `TicketQueryAssistant`
- 预期结果：
  - 返回北京到上海的机票结果


## 2. 交通订单

### TC-010 创建交通订单

- 输入：
  - `帮我预订2026-03-21北京到上海的高铁票，二等座1张`
- 预期路由：
  - `TicketOrderAssistant`
- 预期结果：
  - 成功创建 1 笔高铁订单
  - `orders.status='booked'`

### TC-011 查询当前订单

- 输入：
  - `查询当前订单`
- 预期路由：
  - `TicketOrderAssistant`
- 预期结果：
  - 返回当前用户所有已预订订单
  - 已支持 `train / flight / hotel` 混合展示

### TC-012 多类型订单查询

- 输入：
  - `查询我的飞机票和酒店和火车票`
- 预期路由：
  - `TicketOrderAssistant`
- 预期结果：
  - 不应拆成 `my_orders + hotel`
  - 应统一查询多类型订单

### TC-013 退票

- 前置条件：
  - 当前存在一笔已预订交通订单
- 输入：
  - `帮我退掉2026-03-21北京到上海的高铁票`
- 预期路由：
  - `TicketOrderAssistant`
- 预期结果：
  - 原订单状态改为 `cancelled`
  - 对应库存回补

### TC-014 改签

- 前置条件：
  - 当前存在一笔已预订交通订单
- 输入：
  - `把我2026-03-21北京到上海的高铁票改签到2026-03-22二等座`
- 预期路由：
  - `TicketOrderAssistant`
- 预期结果：
  - 原订单状态改为 `changed`
  - 新订单状态为 `booked`
  - 原库存回补，新库存扣减


## 3. 酒店订单

### TC-020 查询酒店

- 输入：
  - `查询2026-03-21上海的酒店`
- 预期路由：
  - `HotelAssistant`
- 预期结果：
  - 返回上海 `2026-03-21` 的酒店库存
  - 回复中出现预算画像排序说明

### TC-021 预订酒店

- 输入：
  - `帮我订2026-03-21上海外滩云际酒店的高级大床房，住2晚1间`
- 预期路由：
  - `HotelAssistant`
- 预期结果：
  - 成功创建酒店订单
  - 房价展示应与逐晚价格一致，不应误写成固定单晚价

### TC-022 取消酒店

- 前置条件：
  - 当前存在一笔已预订酒店订单
- 输入：
  - `取消我订的2026-03-21上海外滩云际酒店`
- 预期路由：
  - `HotelAssistant`
- 预期结果：
  - 不应误识别成 `cancel_order`
  - 订单状态改为 `cancelled`
  - 库存回补

### TC-023 改期酒店

- 前置条件：
  - 当前存在一笔已预订酒店订单
- 输入：
  - `把我2026-03-21上海外滩云际酒店改到2026-03-22`
- 预期路由：
  - `HotelAssistant`
- 预期结果：
  - 原订单状态改为 `changed`
  - 新订单状态为 `booked`
  - 原库存回补，目标库存扣减

### TC-024 查询酒店订单

- 输入：
  - `查询我的酒店订单`
- 预期路由：
  - `TicketOrderAssistant`
- 预期结果：
  - 统一从订单域查询
  - 不应误走酒店库存查询


## 4. 多轮补参

### TC-030 交通缺出发地

- 输入：
  - `帮我查去上海的高铁票`
- 预期结果：
  - 先追问出发城市
  - 不应自动把 `home_city` 直接写入查询

### TC-031 酒店取消缺选择条件

- 输入：
  - `取消我的酒店`
- 预期结果：
  - 返回缺少酒店名/入住日期等追问
  - `pending_context.domain='hotel'`

### TC-032 travel_plan 缺出发地

- 输入：
  - `结合2026-03-21上海开始两天的天气、交通和酒店，帮我做一个出行方案`
- 预期结果：
  - 追问是否从 `home_city` 出发
  - `pending_context.domain='travel_plan'`
- 继续输入：
  - `从北京出发`
- 预期结果：
  - 不应重复追问出发地
  - 应继续执行 `travel_plan` 子图


## 5. travel_plan 联动规划

### TC-040 纯联动规划

- 输入：
  - `从北京出发结合2026-03-21上海开始两天的天气、交通和酒店，帮我做一个出行方案`
- 预期路由：
  - `WeatherQueryAssistant`
  - `TicketQueryAssistant`
  - `HotelAssistant`
- 预期结果：
  - 返回天气判断、出行建议、票务结果、住宿建议、酒店结果
  - 回复首段应包含 `当前行程状态：...`

### TC-041 travel_plan 识别已有交通订单

- 前置条件：
  - 已存在 `2026-03-21 北京 -> 上海` 的交通订单
- 输入：
  - `从北京出发结合2026-03-21上海开始两天的天气、交通和酒店，帮我做一个出行方案`
- 预期结果：
  - `当前行程状态` 里应说明交通已安排
  - 票务结果应显示 `已有关联交通订单`
  - 不应再重复查一遍交通票并当成未下单推荐

### TC-042 travel_plan 识别已有酒店订单

- 前置条件：
  - 已存在与本次行程日期区间重叠的酒店订单
- 输入：
  - `从北京出发结合2026-03-21上海开始两天的天气、交通和酒店，帮我做一个出行方案`
- 预期结果：
  - 应按酒店订单与行程区间是否重叠来识别，不只看入住首日
  - 酒店结果应显示 `已有关联酒店订单`

### TC-043 travel_plan 判断住宿部分覆盖

- 前置条件：
  - 本次规划 `2026-03-21` 开始 `2` 天
  - 已有酒店订单仅覆盖其中 1 晚或只覆盖后半段
- 输入：
  - `从北京出发结合2026-03-21上海开始两天的天气、交通和酒店，帮我做一个出行方案`
- 预期结果：
  - `当前行程状态` 必须明确说明住宿是“部分覆盖”
  - 要提示还缺哪一晚或还缺几晚
  - 不能误判成“住宿尚未安排”或“住宿已齐备”

### TC-044 travel_plan 交通和住宿都齐备

- 前置条件：
  - 交通与酒店订单均已覆盖本次行程
- 输入：
  - `从北京出发结合2026-03-21上海开始两天的天气、交通和酒店，帮我做一个出行方案`
- 预期结果：
  - `当前行程状态` 应明确说明交通和住宿已齐备
  - `ticket_query` 与 `hotel_query` 允许为空
  - 不应因空 query 触发 schema 校验失败


## 6. 数据库核验建议

建议在关键订单用例后做数据库核验：

- 交通订单
  - `orders.status`
  - 对应 `train_tickets` / `flight_tickets` 余票
- 酒店订单
  - `orders.status`
  - `hotel_room_inventory.remaining_rooms`
- travel_plan
  - 重点核验文案与已有订单是否一致
  - 重点核验“部分覆盖”判断是否正确


## 7. 当前重点风险

- `travel_plan` 已进入第一版可用状态，但仍依赖 LLM 对“覆盖是否足够”的判断，文案一致性还需要更多回归验证。
- `pending_context` 目前仍是会话级轻量状态，不支持跨重启恢复。
- 当前还没有自动化测试框架覆盖上述场景，这份文档主要用于手工回归与后续自动化整理。
