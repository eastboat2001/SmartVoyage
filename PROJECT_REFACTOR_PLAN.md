# SmartVoyage 项目详细改造方案

这份文档用于规划 SmartVoyage 后续改造方向，目标是把当前项目从“可运行的多 Agent Demo”提升为“更贴近真实业务、适合求职展示的智能旅游系统”。

文档按优先级拆分为多个阶段。  
原则是：先做最能提升项目价值、又最容易在面试中讲清楚的部分，再逐步扩展功能边界。

## 当前落地进度

- P0 已完成最小落地
  - 已移除演唱会票主流程与测试数据
  - 已将项目主线收敛到天气、交通票务、订单
- P1 已完成最小落地
  - 已新增 `users`、`orders`
  - 已支持订票后落库
  - 已支持查询我的订单
  - 已支持基于同一用户、同一车次/航班、同一席位的防重复下单
- P1 仍有可继续增强的点
  - 目前为默认演示用户模式，不是真实登录系统
  - 重复下单时当前返回拦截文案，尚未进入“继续下单 / 改签 / 退票”的分支流程
  - 状态一致性已覆盖部分高频查询，但还不是完整订单生命周期管理
- P2 及之后阶段尚未开始


## 一、当前项目现状总结

### 当前已经具备的能力

- 基于 `A2A + MCP + LangChain v1.x`
- 支持天气查询、交通票务查询、交通票务预定
- 支持 `openai_compatible` / `ollama`
- 已经有统一编排层
- 已经有 `travel_plan` 跨 Agent 协作链路
- 已经有基本的失败恢复和降级能力
  - Agent 超时兜底
  - 结构化输出重试
  - fallback provider
  - 无数据和服务失败区分

### 当前最主要的问题

1. 项目定位还不够收敛  
虽然叫“智能旅游项目”，但现有功能里混入了演唱会票，主线不够聚焦，应直接移除。

2. 缺少真实业务状态  
现在系统能查票、订票，但没有用户系统、订单系统、偏好系统，因此无法防重复下单，也无法做改签/退票。

3. 旅游闭环还不完整  
目前主要是天气 + 交通票务，还缺酒店、路线规划、订单管理等更像旅游系统的能力。

4. 评测和演示材料还不够完整  
现在已经能跑，但还缺“稳定演示用例 + 项目亮点叙事 + 评测设计”。


## 二、改造总目标

把项目逐步收敛成一个更完整的智能旅游助手，核心方向为：

- 用户画像
- 天气感知
- 交通方式推荐
- 交通票务查询与预订
- 酒店预订
- 路线/行程规划
- 订单查询 / 退票 / 改票

最终希望项目具备以下特点：

- 有明确的产品主线，而不是功能拼接
- 有真实业务状态，而不是只靠对话历史
- 有多智能体协作，而不是简单路由
- 有异常处理和降级能力
- 有可重复演示的数据和稳定话术


## 三、改造优先级总览

建议按下面顺序推进：

### P0：项目收敛与基础修正

目标：统一项目方向，避免越改越散。

### P1：用户系统 + 订单系统 + 防重复下单

目标：补齐真实业务状态，让订票不再是一次性动作。

### P2：订单生命周期能力

目标：支持“查询我的订单 / 退票 / 改票”。

### P3：用户偏好与画像

目标：让系统做个性化推荐，而不是只看天气。

### P4：酒店能力接入

目标：让项目更像“智能旅游系统”而不是“智能查票系统”。

### P5：路线规划 / 行程规划

目标：增强多智能体协作价值，体现综合规划能力。

### P6：评测体系、展示材料、README 收尾

目标：把项目变成可以稳定展示、稳定讲解的求职项目。


## 四、详细分阶段方案

## 阶段 P0：项目收敛与基础修正

### 目标

把项目边界收紧到“智能旅游”主线，减少低价值复杂度。

### 建议动作

1. 直接移除演唱会票能力

- 从意图识别、票务查询、订票逻辑、测试数据、README 中直接移除演唱会相关能力
- 推荐测试问题全部围绕天气、交通票、订票、旅游规划
- 保持项目主线聚焦在智能旅游，而不是泛票务系统

2. 明确项目主线

建议把项目重新定义为：

- 天气感知出行助手
- 交通票务推荐与预订
- 酒店与行程协作规划
- 用户状态驱动的旅游订单系统

3. 规范日志和启动体验

- 保持当前日志策略：
  - `logs/app.log`
  - `logs/mcp.log`
  - `logs/a2a.log`
- 保持 `run_all.py` 三种启动方式
- 保持 `run_all.py --with-cli` 先启动后端，再进入 CLI

### 交付结果

- 项目定位清晰
- README、测试、演示问题统一
- 删除和项目主线不一致的功能噪音


## 阶段 P1：用户系统 + 订单系统 + 防重复下单

### 目标

让系统拥有“业务状态”，解决重复订票问题，并为后续改签/退票打基础。

### 为什么优先级最高

因为这是当前最大短板，也是最能提升项目工程感的部分。

### 建议新增表

1. `users`

建议字段：

- `id`
- `username`
- `phone`
- `default_departure_city`
- `created_at`

2. `user_preferences`

建议字段：

- `id`
- `user_id`
- `transport_preference`
- `seat_preference`
- `cabin_preference`
- `budget_level`
- `prefer_direct`
- `prefer_morning_departure`
- `created_at`
- `updated_at`

3. `orders`

建议字段：

- `id`
- `user_id`
- `order_type` (`train` / `flight` / `hotel`，后续可扩展)
- `status` (`booked` / `changed` / `cancelled`)
- `departure_city`
- `arrival_city`
- `departure_time`
- `ticket_or_room_type`
- `transport_no`
- `quantity`
- `unit_price`
- `total_price`
- `raw_order_payload`
- `created_at`
- `updated_at`

### 功能目标

1. 订票成功后落库  
2. 新请求进入订票前先查当前用户是否已有相似订单  
3. 如果相似订单已存在，则不直接再次下单，而是先追问：

例如：

`您已预订 2026-03-21 北京到上海 G5 次高铁二等座 1 张，是否还需要再订一张，还是改签/退票？`

4. 支持“查询我的订单”

### 代码改造建议

重点改造位置：

- `sql/create_table.sql`
- `sql/insert_data.sql`
- `mcp_server/mcp_order_server.py`
- `a2a_server/order_server.py`
- `utils/orchestrator.py`
- `utils/structured_outputs.py`

### 阶段验收标准

- 用户成功订票后，订单能持久化
- 相同请求再次发起时，不会直接重复下单
- 能查询当前用户已有订单


## 阶段 P2：订单生命周期能力

### 目标

支持“查询订单 / 退票 / 改票”，让项目从“下单”扩展到“完整订单管理”。

### 功能范围

1. 查询我的订单

例如：

- `帮我看看我订了哪些票`
- `查询我3月21日的订单`

2. 退票

例如：

- `帮我退掉我订的3月21日北京到上海的高铁票`

3. 改票 / 改签

例如：

- `把我3月21日北京到上海的高铁票改签到下午`

### 实现建议

1. 先做退票，再做改票  
因为退票逻辑更简单，更适合先跑通。

2. 改票逻辑建议拆成两步：

- 先识别已有订单
- 再重新查符合新条件的票
- 最后更新订单状态

3. 为订单增加状态变更历史

可选新增表：

- `order_operation_logs`

### 代码改造建议

- `mcp_server/mcp_order_server.py`
- `a2a_server/order_server.py`
- 新增订单查询/退票相关工具函数

### 阶段验收标准

- 能查订单
- 能退票
- 能对已有订单执行改票流程


## 阶段 P3：用户偏好与画像

### 目标

让系统的推荐逻辑从“只看天气”升级为“天气 + 用户偏好 + 票务情况”的综合决策。

### 为什么重要

这是把项目从“会查数据”变成“会做个性化决策”的关键一步。

### 画像建议包含

- 偏好高铁还是飞机
- 预算优先 / 时间优先 / 舒适优先
- 座位 / 舱位偏好
- 是否偏好直达
- 是否偏好早班 / 晚班

### 功能目标

例如面对同一个问题：

`帮我判断从北京去上海更适合坐高铁还是飞机`

不同用户可以得到不同建议：

- A 用户：预算敏感、偏好高铁
- B 用户：时间优先、偏好飞机

### 技术实现建议

1. 在编排层读取用户偏好  
2. 在 `travel_plan` 规划 Prompt 中加入用户偏好上下文  
3. 在推荐理由中体现偏好对决策的影响

### 阶段验收标准

- 系统可以基于不同用户偏好给出不同推荐
- 推荐理由中体现天气与偏好的共同作用


## 阶段 P4：酒店能力接入

### 目标

扩展旅游闭环，让系统不再只围绕交通票务。

### 为什么是这一阶段做

酒店是智能旅游项目里比演唱会更合理的扩展方向，而且和天气、行程、路线都能协同。

### 建议新增内容

1. 酒店表

例如：

- `hotels`
- `hotel_rooms`

2. 酒店 MCP 服务

例如：

- `mcp_server/mcp_hotel_server.py`

3. 酒店 A2A Agent

例如：

- `a2a_server/hotel_server.py`

### 功能目标

- 查询某地某日酒店
- 根据预算、偏好推荐酒店
- 预订酒店
- 查询酒店订单

### 阶段验收标准

- 能查询酒店
- 能预订酒店
- 能把酒店和交通规划串联起来


## 阶段 P5：路线规划 / 行程规划

### 目标

真正体现“多智能体为什么存在”，把项目升级为行程协作系统。

### 目标场景

例如：

`我想3月下旬去上海玩两天，帮我结合天气、交通和酒店做一个方案`

### 规划链路建议

可由编排层驱动以下协作：

1. 天气 Agent：判断天气情况  
2. 交通 Agent：推荐高铁或飞机  
3. 酒店 Agent：给出住宿方案  
4. 行程 Agent：给出每日安排建议  

### 输出形式建议

不是单一查询结果，而是一份结构化方案，例如：

- 出发建议
- 交通建议
- 住宿建议
- 每日路线建议
- 总预算估算

### 阶段验收标准

- 支持跨天气、交通、酒店的协作规划
- 输出一份完整的旅游方案
