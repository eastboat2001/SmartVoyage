# SmartVoyage 重构计划

## 1. 当前状态

### 已完成

- P0：项目主线已收敛到智能旅游，不再保留演唱会票主流程
- P1：用户系统、订单表、防重复下单、查询我的订单
- P2：退票、改签、订单状态流转、库存回补 / 扣减
- P3：`user_preferences`、`home_city` 追问、用户画像接入 `travel_plan`
- P4：酒店查询、预订、取消、改期、酒店订单查询、预算画像排序
- 状态统一第一版：订单域 / 酒店域 / `travel_plan` 已统一到结构化 state 模式
- P5 第一版：`travel_plan` 已支持天气 + 交通 + 酒店联动，并能识别已有订单、判断覆盖缺口
- P6 第一版：已接入 LangSmith 回归骨架，已沉淀核心无副作用回归基线

### 当前仍存在的限制

- `pending_context` 仍是会话级轻量状态，不是持久化 LangGraph 状态
- 多类型订单过滤仍有少量后端归一规则
- `travel_plan` 已支持条件式联动订票，但还没有完整的一键联动酒店下单和更细的每日行程规划
- 用户画像已接入，但对酒店 / 票务推荐的影响还不够丰富
- 订单操作历史表尚未补齐

## 2. 当前主线

当前最合理的推进方向：

- P4 增强：更丰富的画像排序与酒店推荐
- P5 增强：更完整的交通 + 酒店 + 行程联动规划
- P6 增强：副作用测试集、数据隔离、语义 evaluator

## 3. 分阶段路线

### P4 酒店增强

目标：让酒店能力从“可用”变成“更像真实旅游产品”。

下一步建议：

- 让酒店排序接入更丰富的用户画像，不只看 `budget_level`
- 继续优化酒店联动查询文案与筛选维度
- 视需要补酒店独立脚本与更多回归样例

### P5 行程规划增强

目标：把 `travel_plan` 从第一版联动建议升级成更完整的规划工作流。

下一步建议：

- 继续强化“已有订单识别 + 缺口补足”逻辑
- 支持更细的每日行程建议
- 支持“条件式下单意图 + 后端执行判定”场景下的半自动联动下单
- 让 `travel_plan` 更系统地利用用户画像

### P6 评测体系增强

目标：把当前回归基线升级成可持续维护的评测体系。

当前已完成：

- LangSmith dataset + experiment + evaluator 骨架
- `langsmith_eval/cases.json` 无副作用基线集
- 第一轮核心回归跑通

下一步建议：

- 已新增 `langsmith_eval/side_effect_cases.json` 作为第二批副作用测试集，并加入 `setup_profile + db_assertions`
- `run_langsmith_eval.py` 已支持 side-effect dataset 的独立同步
- side-effect runner 已支持逐条数据库重置、`setup_profile` 前置注单，以及数据库前后快照断言
- 当前下一步更适合继续扩展断言覆盖范围，并补更强的 `travel_plan` 语义 judge
- 固化实验基线，方便后续版本对比

## 4. 当前测试策略

### 第一批：无副作用基线

位置：`langsmith_eval/cases.json`

特点：

- 适合频繁批量运行
- 不修改数据库状态
- 覆盖天气、票务、订单混查、酒店查询、酒店补参、`travel_plan` 缺参 / 恢复 / 覆盖判断

### 第二批：副作用测试

方案文档：`SIDE_EFFECT_TEST_PLAN.md`

数据草案：`langsmith_eval/side_effect_cases.json`

覆盖方向：

- 交通订票 / 退票 / 改签
- 酒店预订 / 取消 / 改期
- `travel_plan` 条件式下单意图触发联动订票

推荐隔离策略：

- 优先采用“每轮前完整重建数据库”

## 5. 最推荐的下一步

如果继续推进代码，优先级建议如下：

1. 丰富用户画像对酒店 / 交通推荐的影响
2. 扩展 `travel_plan` 到更完整的交通 + 酒店 + 行程联动方案
3. 基于 `SIDE_EFFECT_TEST_PLAN.md` 新建第二批 LangSmith 副作用测试集
4. 视需要补订单操作历史与更自然的退票 / 改签多轮交互

## 6. 项目一句话定位

把 SmartVoyage 从“可运行的多 Agent Demo”，升级为“具备用户状态、订单生命周期、个性化推荐、跨服务协作与评测基线的智能旅游助手”。
