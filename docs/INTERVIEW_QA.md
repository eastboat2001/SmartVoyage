# SmartVoyage 面试 Q&A 题库

## 使用说明

这份题库按“面试官问一句，你答一段”的方式组织，回答尽量围绕真实系统流程展开，而不是单独堆概念。

回答原则：

- 先讲 Agent 负责什么，再讲它怎么调用工具、怎么组织 Prompt、为什么这样拆
- `MCP` 不单独炫概念，而是放回具体 agent 的工具边界里解释
- `Skill` 不单独说成标准协议，而是放回具体 agent 的任务流程里解释
- 每次回答都尽量落到真实模块、真实收益和真实取舍上

---

## 一、项目整体

### Q1：你先整体讲一下这个项目是怎么做的。

这个项目叫 SmartVoyage，是一个面向交通票务场景的多代理 Agent 系统。它支持天气查询、时间查询、火车票和机票查询，以及下单、查单、退票、改签这些事务型操作。我的核心目标不是做一个普通聊天机器人，而是把“查询、决策、执行”这三类任务放在一个可控的工程体系里。

整体结构上我用了一个 `Supervisor` 和两个本地子代理。`SmartVoyageSupervisor` 负责意图识别、路由和复合任务编排；`TravelReadSubagent` 负责时间、天气和票务只读查询；`OrderSubagent` 负责下单、查单、退票、改签以及 HITL 审批恢复。每个 agent 在做事时都会用到两类能力：一类是模型侧的任务规则，也就是本地 Skill Runtime 装配出来的 Prompt；另一类是外部能力边界，也就是通过 MCP 暴露出来的工具。订单状态流转则交给 LangGraph 处理。后面我又做了性能优化和回归评测，包括 Redis 缓存、deterministic formatting、阶段级 metrics、轻重模型路由和 LangSmith 回归。

### Q2：这个项目最想解决的核心问题是什么？

核心问题是，如何把一个窄领域里的 Agent 系统做成“可控的事务型系统”，而不是只会生成自然语言。因为交通票务场景里有两类完全不同的任务：一类是只读查询，比如天气和票务；另一类是有副作用的事务，比如下单、退票、改签。前者更关注低时延和低 Token，后者更关注准确性、状态管理和人工审批。所以这个项目本质上是在验证：多代理编排、工具边界、任务流程复用、状态机、性能优化和回归评测能不能在一个真实业务链路里配合起来。

### Q3：项目里最有技术含量的点是什么？

我觉得最有技术含量的不是某一个功能，而是把这个项目做成了一个有工程闭环的 Agent 系统。具体来说，我不是只实现了“能查票、能下单”，而是把它拆成了清晰的分层：Supervisor 编排、子代理分工、MCP 工具边界、Skill Runtime、LangGraph 状态机、模型路由和 observability。然后我又补了完整的评测闭环，能用 LangSmith 去看 route match、semantic match、P50/P95/P99 和 Token 变化。也就是说，这个项目不是 Demo 式功能堆叠，而是一个可优化、可解释、可回归的系统。

## 二、为什么这样拆 Agent

### Q4：为什么是一个 Supervisor 加两个子代理？

我没有按功能名词继续拆成很多 agent，而是按执行语义拆成两个子代理。`TravelReadSubagent` 处理只读查询，`OrderSubagent` 处理有副作用的事务操作。这样拆的原因是，这两类任务的风险模型完全不一样。只读链路可以缓存、可以下沉轻模型，也更适合做低时延优化；事务链路则需要状态机、审批、恢复和更强的准确性保证。如果把它们混在一个 agent 里，Prompt 很容易膨胀，而且优化目标会相互冲突。当前项目的业务域还比较聚焦，所以两子代理是复杂度和可维护性的平衡点。

### Q5：为什么不直接做成单 Agent？

如果做成单 Agent，前期实现会更快，但后面会有三个明显问题。第一，查询和事务操作混在一起，Prompt 和上下文会越来越重。第二，有副作用的操作缺少清晰边界，很难做风险控制。第三，性能问题和错误定位会越来越难，因为所有逻辑都耦在一个地方。我的做法是把“判断做什么”放在 Supervisor，把“只读”和“事务”分给不同子代理，这样后面做 metrics、缓存、轻重模型路由和 HITL 都更清楚。

### Q6：为什么不拆成更多 agent，比如天气 agent、票务 agent、订单查询 agent、票务操作 agent？

对更大的旅行平台来说，这样拆是有道理的，但对当前项目不一定更优。因为当前项目复杂度的核心不是业务域数量，而是订单事务状态机。票务查询和订单事务高度相关，如果再拆成更多 agent，会增加 agent hop、上下文传递和编排复杂度。业务少的时候，过度拆分会制造复杂度；业务多的时候，拆分才是在管理复杂度。当前这个项目主要是“查”和“办”两类能力，所以按只读和事务来拆更合理。如果后面加入酒店、路线规划、景点推荐、攻略检索等新域，我才会考虑进一步细拆 agent。

## 三、Supervisor 怎么讲

### Q7：Supervisor 具体是怎么做的？

Supervisor 的职责是意图识别、路由和复合任务编排。它先根据用户输入判断当前请求属于天气、时间、票务、订单还是 `transport_decision`，然后决定该调用哪个子代理。对于简单请求，它只做一次路由；对于复杂请求，比如“高铁还是飞机更合适并可自动下单”，它会先调 TravelRead 查天气，再根据结构化 `ticket_plan` 查票务，再做 decision plan，最后在满足条件时把请求交给 OrderSubagent 进入下单链路。Supervisor 的关键设计点是，它不直接承担底层工具访问，而是只承担编排职责，这样边界清晰，也方便我做 phase metrics。

### Q8：Supervisor 里用了什么 Skill，为什么要用？

Supervisor 里主要用了两类 Skill。第一类是 `intent-routing`，用于意图识别，它负责把用户输入转换成结构化意图和 user query。第二类是 `transport-decision`，用于复合决策规划，比如结合天气结果和用户查询生成高铁还是飞机的建议，以及结构化的 ticket plan。这里用 Skill 的原因不是为了把 Prompt 外包出去，而是为了把“意图识别规则”和“交通决策规则”从业务代码中抽离出来。Supervisor 本身只负责编排，如果规则也写死在代码里，后面优化和删旧逻辑会非常混乱。

### Q9：Supervisor 里有没有用到 MCP？

Supervisor 本身不直接调底层数据库或外部服务，它主要通过调用子代理间接使用 MCP。也就是说，Supervisor 的重点不是“连外部系统”，而是“决定什么时候调用哪个 Agent，再由那个 Agent 走它自己的工具边界”。这样做的好处是，Supervisor 不会耦合到 MySQL、Redis 或订单写操作逻辑，职责更纯粹。面试里我会强调，Supervisor 这一层更多是在做 orchestration，而不是直接做 tool execution。

### Q10：为什么 Supervisor 里不用一个大 Prompt 一次做完？

因为这样做虽然看起来简单，但非常不利于可控性。像 `transport_decision` 这种流程，本质上是“先查天气，再查票，再决策，必要时下单”，它天然是多阶段流程。如果全塞进一个 Prompt，第一很难做工具边界隔离，第二很难插入 metrics，第三性能优化也很难做。所以我更倾向于让 Supervisor 做显式编排，而不是把所有逻辑压成一次大模型推理。

## 四、TravelReadSubagent 怎么讲

### Q11：TravelReadSubagent 是怎么做的？

TravelReadSubagent 主要负责无副作用只读查询。收到请求之后，它先用 `read_kind` 判断当前是时间、天气还是票务查询；如果是天气或票务，还会继续生成结构化 query plan，然后调用 TravelRead MCP 获取结构化数据，最后用 deterministic formatting 输出结果。这里的重点是，它已经尽量避免再做纯总结型 LLM 调用，因为这类调用对信息量没有本质帮助，反而增加时延和 Token。它的设计目标是把只读链路做成低风险、低时延、可缓存、可观测的部分。

### Q12：TravelReadSubagent 里用了什么 Skill，为什么要用？

TravelReadSubagent 里主要用了 `travel-read` 这个 Skill。它包含三个核心 capability：`read_kind`、`weather_plan` 和 `ticket_plan`。`read_kind` 用于判断当前请求是查时间、天气还是票务；`weather_plan` 和 `ticket_plan` 负责把用户输入转成结构化查询计划。这里之所以用 Skill，是因为这些任务非常依赖稳定的结构化规则，比如相对日期怎么处理、什么时候需要补槽、什么时候可以直接查。如果把这些规则写在业务代码和 Prompt 字符串里，维护会很快失控。Skill Runtime 让这些规则以资产方式复用，TravelReadSubagent 只管调用。

### Q13：TravelReadSubagent 里用到什么 MCP，为什么要这样做？

TravelReadSubagent 会通过 `TravelRead MCP` 来访问当前时间、天气和票务数据。这里我用 MCP 的原因，是想把只读工具能力变成清晰的边界：Agent 不直接知道数据库表结构，不直接知道 Redis key 设计，也不直接拼接底层查询逻辑。它只知道“我要查天气”或者“我要查票”，然后通过 MCP 调用对应工具。这样做的收益是，编排层干净、缓存层集中、工具可复用，而且更容易测试。

### Q14：TravelRead MCP 里具体有哪些工具？

主要有三类工具。第一类是 `get_current_time`，返回当前时区下的时间和日期。第二类是 `query_weather`，根据城市和日期范围查天气。第三类是 `query_tickets`，根据车次或航班号、或者路线加日期查票务。除此之外，这个 MCP 内部还负责把只读缓存接到 Redis 上。也就是说，TravelRead MCP 不是一个泛化平台，而是针对 TravelReadSubagent 这条只读链路设计的外部能力边界。

### Q15：为什么 TravelRead 这条链路适合缓存？

因为它全是无副作用查询。当前时间、天气和票务查询本身都不会修改业务状态，而且在短时间内重复度比较高，所以适合做缓存。更重要的是，我把缓存放在 MCP 这一层，而不是 Agent 里。这样上层 Agent 看到的还是统一工具接口，缓存策略对编排层透明，同时 metrics 也能记录 cache hit 和 miss，便于验证收益。

### Q16：为什么 Weather 和 Ticket 的 summary 不继续用 LLM？

因为在这个项目里，天气和票务查询返回的数据结构已经足够清晰，继续再调用一次模型做“总结”，大多数时候只是把事实重述一遍，收益很低，但会显著增加时延和 Token。所以我把它改成 deterministic formatting，让 TravelReadSubagent 自己按模板输出。这个优化的价值非常直接：减少模型调用、降低时延、提高结果稳定性，而且回归也更容易做。

## 五、OrderSubagent 怎么讲

### Q17：OrderSubagent 怎么做的？

OrderSubagent 处理所有有副作用的订单事务。它先做动作分类，判断是查询订单、下单、退票还是改签；然后做日期归一化、信息抽取和缺槽判断。如果信息不足，它会通过 pending context 继续追问；如果进入高风险操作，就走 LangGraph 的状态机，并在必要节点进入 HITL。整个过程中，订单状态和中间上下文可以持久化到 checkpoint，这样中断后还能恢复。这个模块最关键的地方是，它不是自由生成文本，而是围绕结构化输出和状态机在运行。

### Q18：OrderSubagent 里用了什么 Skill，为什么要用？

OrderSubagent 里主要用了 `order-operation` 这个 Skill。它包含 `action_classify`、`date_resolution`、`operation_extraction`、`review_decision` 这些 capability。也就是说，订单子代理里的 Skill 不是拿来“聊天”的，而是拿来做动作判定、日期归一化、订单字段抽取和审批判断。之所以用 Skill，是因为订单链路对规则稳定性要求更高，像退票和改签需要的字段、缺槽时怎么追问、审批时怎么表达，这些都更适合做成可复用流程资产，而不是散在代码里。

### Q19：OrderSubagent 里用了什么 MCP，为什么要这样做？

OrderSubagent 会通过 `Order MCP` 去做订单创建、订单查询、订单取消、改签以及库存扣减和回补。这里用 MCP 的意义在于，把有副作用的工具调用统一放在一个明确边界后面。Agent 自己不直接拼数据库写操作，也不直接控制库存更新，而是通过 MCP 进行调用。这样做能把业务编排和事务执行分开，也更容易解释哪部分是“模型负责理解和规划”，哪部分是“系统负责严格执行”。

### Q20：Order MCP 为什么不能像 TravelRead MCP 一样缓存？

因为订单链路是有副作用和强状态的。下单、退票、改签本质上都在修改订单和库存状态，即使是查询订单，也可能依赖最新事务结果。如果在这一层缓存，很容易带来一致性问题。所以我的策略是：只读查询才缓存，有副作用事务不缓存。面试时这点其实很好讲，因为它直接体现了我对系统正确性边界的控制。

### Q21：OrderSubagent 为什么要引入 LangGraph 和 HITL？

因为订单链路不是单步问答，而是状态机。用户可能先说“帮我改签”，系统要先判断是哪张订单、改到什么时候、有没有缺槽、是否需要审批、审批通过后再真正执行。这种流程天然适合用状态机建模，而不是靠一次大模型生成全部结果。HITL 的作用则是兜住高风险节点，比如真正执行有副作用操作前，需要人工确认。LangGraph 和 HITL 放在一起，本质上是为 OrderSubagent 提供一个可恢复、可中断、可审核的执行框架。

## 六、transport_decision 怎么讲

### Q22：transport_decision 为什么值得单独讲？

因为它是这个项目里最能体现“查询 + 决策 + 执行”组合价值的流程。用户不是单纯查天气，也不是单纯查票，而是问“在当前天气和票务条件下，坐高铁还是飞机更合适”。这个流程要求系统先查天气，再查票，再做建议，有时还要继续下单。所以我把它作为一个专门的复合流程来设计和优化。实际优化时我也发现，它是整个系统里最容易发生 LLM 往返膨胀的地方，所以性能优化基本都是围绕它展开的。

### Q23：transport_decision 里具体用了哪些 agent、skill 和工具？

这个流程是由 Supervisor 主导编排的。首先，Supervisor 用 `intent-routing` Skill 做意图识别；然后把天气查询交给 TravelReadSubagent，TravelRead 再用 `travel-read` Skill 生成天气查询计划，并通过 TravelRead MCP 获取天气数据；接着 Supervisor 用 `transport-decision` Skill 生成结构化的决策结果和 ticket plan；再把票务查询继续交给 TravelReadSubagent，通过 TravelRead MCP 查票；最后 Supervisor 汇总成出行建议，必要时再把请求交给 OrderSubagent 进入下单流程。也就是说，这个流程能很好地展示“编排层、任务流程资产、工具协议边界”三者是怎么配合的。

### Q24：你对 transport_decision 做了哪些优化？

我最先定位到的瓶颈就是这个流程，因为它容易触发多次模型调用。后面我做了几件事。第一，合并重复决策，去掉独立的 `auto_order` 判定，把是否下单收敛到 `decision_plan`。第二，把天气和票务结果总结改成 deterministic formatting，减少纯总结型模型调用。第三，用结构化 `ticket_plan` 直接驱动查票，避免自然语言 query 再被二次解析。第四，给只读工具加 Redis 缓存。第五，增加 phase-level metrics，精确看到是 `intent_recognition`、`weather_plan`、`ticket_plan` 还是 `decision_plan` 在耗时。这样优化后，`transport_decision` 的时延和 Token 都明显下降。

## 七、测试和优化怎么讲

### Q25：你怎么测试这个项目？

我把测试分成四层。第一层是单测，主要测结构化输出、标签处理、deterministic formatter 和模型路由这些纯逻辑。第二层是子代理测试，验证 TravelRead 和 Order 的输入输出协议、追问逻辑和 pending context。第三层是集成测试，覆盖 MCP、缓存、订单状态流和 fallback。第四层是 LangSmith 回归评测，用固定 case 集看 route match、semantic match、时延分位数和 Token 消耗。这样我不是只验证“功能能不能跑”，而是同时验证“功能是否回归”和“优化是否真的生效”。

### Q26：你怎么测试 Agent、Skill 和 MCP 这三层？

我的做法是分层测，而不是混着测。Agent 层主要看输入输出协议、路由和状态流；Skill 层主要看当前 `role + capability` 能不能正确装配出 Prompt，以及规则资产是否一致；MCP 层主要看工具边界，包括只读查询结果、缓存命中、订单事务行为和异常降级。这样分层测试的好处是，出了问题能比较快定位到底是编排逻辑错了、Prompt 流程资产错了，还是底层工具边界有问题。

### Q27：你是怎么做性能优化的？

我先通过 metrics 和 LangSmith trace 找瓶颈，发现主要问题不在 MySQL，而在冗余 LLM 往返，尤其是 `transport_decision`。所以后面的优化基本围绕减少不必要模型调用展开。具体做了几件事：合并重复决策、去掉独立 `auto_order` 判定、把天气和票务总结改成 deterministic formatting、给只读链路加 Redis 缓存、补 phase-level metrics，然后对低风险结构化阶段做轻重模型分流。整个优化过程是“先观测，再改，再回归验证”，而不是直接猜哪里慢。

### Q28：为什么你敢把部分任务下沉到轻模型？

因为我没有把所有阶段都下沉，而是只挑那些输入短、输出结构固定、可以做 schema 校验、失败可 fallback 的阶段，比如 `intent_recognition`、`weather_plan`、`ticket_plan`、日期归一化。相反，像 `decision_plan`、复杂订单抽取、最终执行判断，我是保留在强模型上的。所以我不是把“便宜模型”硬塞进系统，而是按风险和可校验性来分层。

### Q29：如果轻模型出错了怎么办？

我的策略是四层兜底。第一，只让低风险结构化任务走轻模型；第二，轻模型输出先过结构校验；第三，失败时自动回退到主模型；第四，用回归集持续验证边界。一旦某个阶段在回归里出现明显波动，我就把它移回强模型。也就是说，这套策略不是拍脑袋，而是有白名单、有 fallback、有回归约束的。

## 八、项目价值和边界

### Q30：这个项目会不会太浅？

我不会把它说成“业务很大而全”，因为那不真实。更准确的说法是，这个项目业务范围是刻意收敛的，但技术深度不浅。它不是做一个大而泛的旅行平台，而是在窄领域里把多代理编排、事务状态机、HITL、Skill Runtime、MCP、性能优化和评测闭环都打通了。所以它更像是一个窄领域深打的 Agent 系统，而不是一个宽业务但很浅的功能 Demo。

### Q31：如果让你继续做下一步，你会怎么演进？

我会沿两个方向演进。第一是业务域扩展，如果后续加入酒店、路线规划、景点推荐、攻略检索，就会考虑再细拆更多 agent。第二是平台能力增强，比如更细的模型路由、更系统的 trace 分析、更稳定的评测流水线，以及更完善的观测面板。但在当前阶段，我不会为了“看起来复杂”而过度拆架构，因为现阶段的重点是稳定性、可解释性和可评测性。

## 九、可直接背诵的总结句

### 总结句 1

这个项目最重要的不是做了多少功能，而是我把一个窄领域 Agent 系统做成了可分层、可优化、可回归验证的工程化系统。

### 总结句 2

我对 MCP 和 Skill 的理解是放回 Agent 里讲的：MCP 负责给 Agent 提供清晰的外部能力边界，Skill 负责给 Agent 提供可复用的任务流程和提示资产。

### 总结句 3

两子代理不是能力不够，而是当前业务范围下复杂度和可维护性的平衡点；如果业务域扩展，再细拆更多 agent 才更有价值。
